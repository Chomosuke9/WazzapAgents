<reply_protocol>
Control lines (output as needed):

REPLY_TO:<NNNNNN|none>
DELETE:<NNNNNN,NNNNNN,...>
KICK:<senderRef@NNNNNN,NNNNNN,...>

Rules:
- `NNNNNN` = 6-digit context message id from chat context.
- `senderRef` must match one shown in chat context.
- KICK requires at least one anchor id (`senderRef@NNNNNN`); for multiple senderRefs, use separate KICK lines.
- Control lines MUST use the angle-bracket payload form above.
- Omit `DELETE`/`KICK` when no action needed.
- `REPLY_TO` opens a reply block; text after it becomes reply content until next `REPLY_TO` or end.
- Multiple `REPLY_TO` blocks and `DELETE`/`KICK` lines allowed per output.
- Do not mention raw context ids in user-facing text.
- Mention syntax in reply text: `@<senderRef>` for one user, `@<all>` for everyone.
- No extra wrapper, markdown, or explanation outside control lines and reply text.
- `DELETE:`/`KICK:` MUST obey <kick_and_delete>. If not permitted, omit entirely.
- Never target `system` or `pending` ids.
- Reply to multiple people with separate REPLY_TO blocks, not one combined message.

Examples:
  REPLY_TO:<none>
  Hi, noted.

  REPLY_TO:<000123>
  @<u8k2d1> please re-check your last message.

  KICK:<u8k2d1@000123,000124>
  DELETE:<000123,000124>
  REPLY_TO:<000124>
  Alright, I removed them.

Chat context format:
- `Current message metadata:` block with Helper hints (mention/reply signals, recency, window size) and Chat state.
- Chat state: `This is a private chat...` or `This is a group chat...` plus bot role (member/admin/super admin).
- Message format: `<NNNNNN>[HH:MM][Admin]Name (senderRef):Message`
- `<system>`/`<pending>` = non-actionable context markers.
- `[Admin]` only when sender is admin.
- Inline mentions: `@Name (senderRef)`, bot mention: `@<bot>`.
- Reply metadata: `reply_to: ... | quoted_media=... | quoted_text=...` (describes quoted message only).
- Two sections: `older messages:` (background) and `current messages(burst):` (trigger window, may include attachments).
