
I need you to improve the LLM1 router prompt in `python/bridge/llm1.py`.

LLM1 is a fast router that reads WhatsApp group chat history and decides 
whether the AI agent should respond. It outputs: should_response (bool), 
confidence (0-100), reason (string).

## Current problems
1. Over-responding: bot joins human-to-human conversations, reacts to 
   emotional moments, keeps following up when not asked
2. Under-responding: bot stays silent when someone asks a question that 
   nobody has answered

## Root cause
The current prompt defaults toward responding and tries to suppress. 
Research on human group chat behavior (Latané & Darley bystander model; 
Sacks et al. turn-taking; Goffman participation framework) shows humans 
default to SILENCE in groups and only respond when specifically addressed 
or uniquely competent. The prompt architecture needs to be inverted.

---

## Changes to make in `build_llm1_prompt()`, inside `base_system`

### Change 1: Add a "core rule" at the very top of the system prompt

Add this as the first line after the agent identity line:

```
Core rule: Default state is SILENT. Respond only when evidence 
clearly justifies it. Being talked ABOUT is not being talked TO. 
An active conversation you were not invited into is not yours to join. 
When in doubt, stay silent — silence is the correct behavior for most messages.
```

WHY: Latané & Darley (1968) showed humans default to non-response in 
groups (only 31% help when 4 others present vs 85% alone). Blair et al. 
(2005) confirmed this digitally. The current prompt has no explicit default 
state, so the LLM treats ambiguous cases as respond-by-default. Inverting 
the default fixes over-responding at the root.

---

### Change 2: Replace the entire "## When to respond" section with a 
4-tier structure

Replace the current "## When to respond" block with:

```
## Response tiers — evaluate in order, stop at first match

**MUST RESPOND** (confidence 90–95):
- Bot is @mentioned (`botMentioned=true`)
- Message is a direct reply to the bot (`repliedToBot=true`)

**SHOULD RESPOND** (confidence 65–80) — only if no human has 
already answered adequately:
- Current window contains a clear question that has gone unanswered 
  for 3+ messages AND the topic is within bot's domain
- Explicit open help request ("ada yang tahu?", "tolong bantu", 
  "anyone know?") with no human response after 3+ messages

**MAY RESPOND** (confidence 40–60) — use careful judgment:
- Bot is in an active thread (`messagesSinceAssistantReply <= 2`) 
  AND the message is a direct follow-up question to the bot's 
  last reply specifically

**MUST NOT RESPOND** — this is the DEFAULT when no tier above matches:
- Two or more humans actively conversing with each other
- Emotional/social content: jokes, reactions, congratulations, 
  venting, grief, excitement — even if it feels "nice" to respond
- Message is a reply to a specific human (not the bot)
- Bot just responded (`messagesSinceAssistantReply <= 1`) and 
  no direct follow-up question to the bot
- Question already answered adequately by a human member
- Greeting or farewell exchanges between humans
- Casual banter, memes, or humor between humans
```

WHY: 
- Tier structure comes from Bhagtani et al. (2025, arXiv 2603.11409) 
  "Speak or Stay Silent" — the first paper to benchmark multi-party 
  turn-taking for AI agents. They classify utterances into I1 (explicit 
  address → must respond), I2 (contextual → should respond), S1 (bystander 
  → must not), S2 (referenced but not addressed → must not).
- The unanswered question trigger implements Latané & Darley's "last resort" 
  pattern: wait for humans first (Step 3: assume others will handle it), 
  then intervene when it's clear nobody will.
- Stivers & Robinson (2006) showed 85% of questions in English receive 
  answers — an unanswered question after 3+ messages is a genuine gap.

### Change 3: Fix "conversation continuity" — currently too broad

Find and replace this rule:
```
Respond (conversation continuity): if the bot recently replied and the 
current messages continue the same topic, follow up on the bot's answer, 
or ask a related question — respond. Do not go silent mid-conversation 
just because there is no explicit @mention.
```

Replace with:
```
Respond (conversation continuity): ONLY if the bot recently replied AND 
the current message is a direct follow-up question specifically to the 
bot's last reply. The topic still being active is NOT sufficient reason 
to respond. If humans have taken over the topic, exit the conversation.
```

WHY: The current rule has no exit condition. Sacks et al.'s (1974) 
turn-taking model shows that once another party takes the floor, the 
prior speaker's obligation ends. MUCA (Mao et al., 2024, arXiv 2401.04883) 
found that managing chime-in rate is critical — "too many responses are 
disruptive." The current rule causes the bot to follow a topic indefinitely, 
which is the primary over-responding failure mode.

### Change 4: Fix "name in text" — add negative examples

Find and replace:
```
Respond (name in text): if the bot's name appears in message text 
(even without @mention), the user is likely talking to or about the bot. 
Lean toward responding unless the context is clearly not directed at the bot.
```

Replace with:
```
Respond (name in text): ONLY if the bot's name appears in a sentence 
directed AT the bot (e.g., "[name], bisa tolong...?", "hey [name] what 
is..."). 

MUST NOT respond if the bot's name appears in third-person reference:
- "tadi si [name] bilang..." → talking ABOUT the bot, not TO it
- "[name] udah jawab itu" → referencing bot's prior message
- "kata [name]..." → quoting the bot in human conversation
These are S2 (referenced, not addressed) per Goffman (1981) — 
ratified bystander, not addressee.
```

WHY: Goffman's (1981) participation framework distinguishes "addressed 
recipient" from "unaddressed recipient." Being mentioned is not the same 
as being addressed. The current rule has no negative examples, so the LLM 
treats any name occurrence as an invitation to respond. Bhagtani et al.'s 
S2 category specifically covers this case.

### Change 5: Define "long silence" numerically

Find and replace:
```
Respond: long silence since last reply
```

Replace with:
```
Respond (gap coverage): if `messagesSinceAssistantReply >= 8` AND the 
latest message is an unanswered question or help request. Do NOT use 
"long silence" as a reason to respond to non-question messages.
```

WHY: The current phrase "long silence" is undefined, so the LLM 
interpolates inconsistently. Avrahami & Hudson (2006), analyzing 90,000+ 
IM messages, found urgency and wait-time are the top predictors of response 
obligation — but only for questions, not for general messages. The threshold 
of 8 corresponds roughly to the group moving past the topic, at which point 
the bot becomes the "last resort" responder per the bystander model.

### Change 6: Cap react-only usage

Find the react-only rule and append:
```
React-only limit: use at most once per distinct emotional event. Do NOT 
react-only to every positive message in a burst. If uncertain whether 
a reaction is warranted, stay silent instead — silence is less disruptive 
than excessive reactions.
```

WHY: The current rule has no frequency cap. MUCA's user studies showed 
excessive bot participation (even non-verbal) increases perceived 
intrusiveness. Nonnecke & Preece (2001) documented that peripheral 
participation is a legitimate, respected mode — the bot should not feel 
compelled to acknowledge every emotional moment.
